# Analysis Framework

这个框架用于指导 agent 分析 GitHub Trending。目标是从热门仓库中提炼真实趋势和行动建议，而不是复述榜单。

## 触发意图

当用户出现以下意图时使用该框架：

- 分析 GitHub Trending、热门仓库、开源趋势、GitHub 榜单
- 分析某个语言或方向的热门项目
- 判断一个 GitHub 仓库是否值得学习、使用、fork、集成、投资关注
- 生成技术趋势日报、周报、月报
- 从开源项目中寻找产品机会、创业机会或学习路线

## 工作流

1. 明确范围  
   确认时间范围、语言、主题和用户角色。用户没有指定时，默认 daily、全语言、中文报告、开发者视角。

2. 获取候选项目  
   按 `core/data-source-strategy.md` 获取数据。保留来源和可信度。无法联网时，请用户提供列表，不编造实时数据。

3. 初筛  
   去掉明显无关、重复、镜像、教程搬运、缺少仓库信息的项目。保留异常高热度项目，但标注风险。

4. 补充上下文  
   对重点项目补充 README、license、release、issue/PR、commit 活跃度、同类项目和生态背景。

5. 评分  
   使用 `core/scoring-rubric.md`。分数必须服务于判断，不要制造虚假精确感。证据不足时给区间或降低置信度。

6. 提炼趋势  
   从项目之间寻找共同模式，例如：
   - 多个项目解决同一类开发痛点
   - 某个新模型、协议、框架带动工具链增长
   - 从 demo 转向生产化基础设施
   - 开源替代闭源服务
   - 个人开发者工具变成团队工作流

7. 输出报告  
   使用 `core/report-templates.md` 中最贴近用户意图的模板。报告必须包含结论、证据、风险和下一步。

## 判断原则

- 区分“热度”和“价值”。Trending 是入口，不是结论。
- 优先看项目解决的问题，而不是只看技术名词。
- 对 AI 项目尤其谨慎：检查可复现性、eval、失败案例、成本和依赖边界。
- 对 DevTools 项目关注安装路径、集成成本、维护活跃度和迁移风险。
- 对基础设施项目关注生态兼容性、性能证据、稳定性和 license。
- 对学习型项目关注知识迁移性，而不是只看 demo 是否酷。

## 角色化输出

如果用户没有指定角色，默认使用开发者视角。若能从问题中推断角色，自动调整：

- “值得学吗”“源码”“技术栈”: 开发者
- “能不能做产品”“商业化”“创业机会”: 创业者 / 产品经理
- “赛道”“趋势”“方向”: 战略 / 投资研究
- “论文”“方法”“benchmark”: 研究者

## 输出质量要求

报告必须做到：

- 开头先给结论
- 对每个重点项目给出建议动作
- 说明数据来源和不确定性
- 至少包含一个风险或反证角度
- 避免堆砌项目列表
- 避免把 README 文案改写成分析

## 禁止行为

- 不要编造“今日 Trending”数据
- 不要编造 star 增量、融资、用户数、benchmark
- 不要把所有热门项目都评价为“值得关注”
- 不要只输出链接和摘要
- 不要忽略 license、安全和维护风险
