# Scoring Rubric

使用 100 分制评估仓库价值。评分不是排名的机械替代，而是帮助 agent 避免只看 star 增长。

## 总分结构

| 维度 | 权重 | 关注点 |
| --- | ---: | --- |
| 趋势强度 | 20 | star 增长、榜单位置、跨社区讨论、时间窗口 |
| 技术价值 | 20 | 技术新意、架构质量、可复用性、同类差异 |
| 工程成熟度 | 20 | README、安装体验、测试、release、issue/PR 维护 |
| 实用价值 | 15 | 解决的问题是否真实，目标用户是否清晰，场景是否高频 |
| 生态位置 | 10 | 是否顺应大趋势，是否能成为基础设施或插件生态 |
| 学习价值 | 10 | 是否值得开发者投入时间学习，能否迁移到其他项目 |
| 风险折扣 | -15 到 5 | 许可证、维护不确定性、安全风险、炒作、过度承诺 |

## 分数解释

- 85-100: 高价值项目，值得重点研究或持续跟踪
- 70-84: 有明确价值，适合学习、试用或加入观察列表
- 55-69: 有亮点但不确定性较高，适合轻量浏览
- 40-54: 短期热点或局部有用，需要谨慎投入
- 0-39: 噪声较大，除非与当前需求强相关，否则不建议投入

## 风险折扣规则

出现以下情况应主动降分：

- README 只有营销描述，缺少安装、示例或限制说明
- star 暴涨但 commit/release/issue 活跃度很弱
- 项目声称替代成熟基础设施，但没有 benchmark 或生产案例
- license 缺失、模糊或与用户目标冲突
- 依赖闭源服务但没有说明数据、安全和成本边界
- AI 相关项目没有说明评测集、失败案例或可复现方式

## 不同用户的权重调整

开发者：

- 提高学习价值、工程成熟度、技术价值权重
- 输出是否值得读源码、上手难度和可迁移知识点

创业者 / 产品经理：

- 提高实用价值、生态位置、商业机会权重
- 输出目标用户、替代方案、商业化可能性和进入窗口

投资 / 战略研究：

- 提高趋势强度、生态位置、同类项目比较权重
- 输出方向判断、赛道拥挤度和领先信号

研究者：

- 提高技术价值、可复现性、创新性权重
- 输出相关论文、方法差异和实验可信度
